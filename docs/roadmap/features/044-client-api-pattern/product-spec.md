# Product Spec: client-api-pattern

**Created**: 2026-05-28

---

## Problem Statement

All three Next.js frontends use SWR for client-side data fetching, but the layer is untyped and
unmaintainable: Connect-RPC service descriptors use `I: {} as any, O: {} as any` placeholders,
mutation calls are plain `fetch()` with no loading/error state, and response shapes leak `any`
into component props. Additionally, SWR lacks automatic cache propagation — when a mutation
updates an entity, every query that displays it must be manually invalidated at the call site,
creating a growing dependency graph as new panels are added.

## User Story

As a frontend developer, I want every client-side API call expressed as a typed hook backed by
TanStack Query + normy so that (1) incorrect request/response shapes are caught at compile time,
(2) mutations automatically propagate entity updates to all co-mounted queries without manual
invalidation bookkeeping.

## Functional Requirements

FR-1. The `*_connect.ts` generated files (from `@connectrpc/protoc-gen-connect-es`) must be
added to the compiled `dist/` output of `packages/proto/gen/ts/`. Currently they are source-only;
adding them to the build makes them importable as
`@xstockstrat/proto/<service>/v1/<service>_connect`.

FR-2. Each frontend service must add `@xstockstrat/proto` as a workspace dependency in its
`package.json`. The three frontends already carry the necessary runtime (`@bufbuild/protobuf`,
`@connectrpc/connect-web`) — only the package reference is missing.

FR-3. The `{} as any` placeholders in `connectClients.ts` / `configClient.ts` must be replaced
by importing the generated service descriptors directly:
```typescript
// Before
placeOrder: { name: 'PlaceOrder', I: {} as any, O: {} as any, kind: MethodKind.Unary }

// After
import { TradingService } from '@xstockstrat/proto/trading/v1/trading_connect'
// TradingService already carries fully typed I/O — no manual descriptor needed
```

FR-4. Request/response types in route handlers and hook files must be imported from
`@xstockstrat/proto/<service>/v1/<service>_pb` (generated by `@bufbuild/protoc-gen-es`) rather
than declared as `any[]` or inline literals. These types are tree-shakeable and stay in sync
with proto automatically on every `buf-gen.sh` run.

FR-5. Replace SWR with `@tanstack/react-query` v5 + `@normy/react-query` across all three
frontends:
- Install `@tanstack/react-query` and `@normy/react-query`; uninstall `swr`
- Wrap each service's root `layout.tsx` in `QueryClientProvider` (from
  `@tanstack/react-query`) and `QueryNormalizerProvider` (from `@normy/react-query`)
- Configure `getNormalizationObjectKey` to cover the domain entity types that are safe to
  normalize (entities with a stable, domain-unique string ID):
  ```typescript
  getNormalizationObjectKey: obj => {
    if ('orderId' in obj && typeof obj.orderId === 'string') return `order:${obj.orderId}`
    if ('strategyId' in obj && typeof obj.strategyId === 'string') return `strategy:${obj.strategyId}`
    return undefined  // positions, config keys, portfolios: too-generic field names — skip
  }
  ```
- The normalizer config must live in a shared `lib/queryClient.ts` file per service so it is
  not duplicated across layout files.

FR-6. Every read operation (data displayed in a component) must be wrapped in a named, typed
`useQuery` hook (e.g. `useOrders`, `usePortfolio`, `useStrategies`) with explicit `TData` and
`TError` generics. Components must not call `useQuery` directly with an untyped fetcher.

FR-7. Every write/mutation operation (POST, PATCH, DELETE) must be wrapped in a named, typed
`useMutation` hook with explicit `TData`, `TError`, and `TVariables` generics. Mutation
`mutationFn` implementations must return the full updated entity (not just `{ success: true }`)
so normy can propagate the update to all co-mounted queries that hold the same entity. Components
must not call `fetch` directly for mutations.

FR-8. `any` must be eliminated from the public surface of all hook return types and
request/response shapes. `unknown` + type guards are permitted internally where a runtime shape
cannot be statically guaranteed (e.g., caught errors).

FR-9. All three services must follow an identical directory layout and naming convention for the
hook layer, documented in a new `docs/patterns/client-api-pattern.md` pattern file. The pattern
file must include: directory structure, `lib/queryClient.ts` template, a `useQuery` example
hook, a `useMutation` example hook, and the normy `getNormalizationObjectKey` extension guide.

FR-10. All existing `useSWR` calls in component files and all bare `fetch()` mutation calls in
component files must be migrated to the named typed hooks. Config-ui's `useEffect` + `fetch`
data-loading pattern must also be migrated to `useQuery`. No direct `useQuery` or `useMutation`
calls in component files after this feature lands.

FR-11. Error handling in all route handlers must use `unknown` instead of `any` in catch
clauses, with a narrow type guard (`instanceof Error`) before accessing `.message`.

## Out of Scope

- SSE/streaming endpoints (`/api/alerts/stream`) — streaming routes follow a separate pattern
  and are excluded from this feature.
- Server-side route handler logic (Connect-RPC call construction) — only the
  client→route-handler boundary is in scope.
- Adding new API endpoints or changing backend proto contracts.
- Expanding `getNormalizationObjectKey` beyond `orderId` / `strategyId` — positions (`symbol`),
  config keys (`key`), and portfolios use field names that are too generic to normalize safely
  without risking cross-entity collisions. Expand in a follow-up once the pattern is proven.
- End-to-end (Playwright) tests — unit/type-level coverage is sufficient.
- `ts-proto` generated stubs (`gen/ts/**/*.ts` without `_pb` / `_connect` suffix) — these are
  gRPC-JS client classes, not used by the frontends.

## Acceptance Criteria

1. `*_connect.ts` files are present in `packages/proto/gen/ts/dist/` and importable as
   `@xstockstrat/proto/<service>/v1/<service>_connect`.
2. `tsc --noEmit` passes with zero errors in all three services after the changes.
3. `grep -rn '{} as any' src/` returns zero hits in all three services — all service
   descriptors use imported `@xstockstrat/proto` types.
4. `grep -rn ': any' src/` (or `./` for config-ui) returns zero hits inside hook files;
   only permitted inside internal type-guard bodies.
5. `grep -rn 'useQuery\|useMutation' src/app src/components` (or `./app ./components` for
   config-ui) returns zero hits — all data fetching goes through named hook wrappers.
6. `grep -rn 'catch (err: any)' src/` returns zero hits across all three services.
7. `grep -rn 'swr' src/` returns zero hits — SWR fully removed.
8. The new `docs/patterns/client-api-pattern.md` pattern file documents the directory
   layout, `lib/queryClient.ts` template, hook naming conventions, and a minimal example
   for each operation type (query, mutation, normy config extension).
9. All existing UI features (order placement, portfolio view, strategy scoring, config
   mutation) continue to function correctly — no regressions in Connect-RPC proxying.

  codebase's use of Protobuf Message types. One-time migration cost accepted.
