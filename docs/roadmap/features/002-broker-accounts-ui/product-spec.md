# Product Spec: broker-accounts-ui

**Created**: 2026-05-06

---

## Problem Statement

The `add-ikbr-account-support` feature added `ListBrokerAccounts`, `ListPortfolios`, `RegisterBrokerAccount`, `DeregisterBrokerAccount`, and account-scoped `PlaceOrder` RPCs to the backend, but explicitly deferred all UI changes. Users can neither manage their broker accounts nor view per-account portfolio data without making raw API calls.

## User Story

As a platform user, I want to register and remove broker accounts from the trader UI, select which account to trade with, and view portfolio and position data scoped to each account, so that I can manage Alpaca and IBKR accounts end-to-end without making API calls or restarting any service.

## Functional Requirements

FR-1. Add an **Account Selector** component to `xstockstrat-trader` mounted in the **global persistent header** (root layout). It fetches the account list from `TradingService.ListBrokerAccounts` on app load and renders each account with its `display_name`, `broker_type` (Alpaca / IBKR badge), and `is_paper` indicator. Placing it in the global header means a single React context instance serves all pages without prop drilling or per-page fetch duplication.

FR-2. The currently selected account is stored in React context at the root layout level and persists across page navigations within the session. All trade and portfolio calls pass the selected `account_id`.

FR-3. The **Place Order** form passes the selected `account_id` in `PlaceOrderRequest`. When no account is selected, the form is disabled with a prompt to select an account.

FR-4. The **Portfolio** panel calls `PortfolioService.ListPortfolios` and displays a tab or card per account, each showing `account_id`, `broker_type`, and the positions/P&L for that account. The existing single-account portfolio view becomes the per-account view.

FR-5. The **Positions** table is scoped to the selected account via the optional `account_id` field on `ListPositionsRequest`. When the user selects a different account, the table re-fetches.

FR-6. Account cards show `is_active` state; inactive (deregistered) accounts are visually dimmed and excluded from the account selector's active list.

FR-7. `xstockstrat-insights` analytics dashboard shows a per-account portfolio selector using `ListPortfolios`. On initial load it defaults to the **first account** returned by the list. An **"All Accounts"** option is always available in the selector; when chosen, the dashboard aggregates data client-side across all accounts. The selected account persists in URL state so deep links work correctly.

FR-8. No new RPCs or proto messages are introduced — all RPCs (`ListBrokerAccounts`, `ListPortfolios`, `RegisterBrokerAccount`, `DeregisterBrokerAccount`) exist in the stubs generated by `add-ikbr-account-support`.

FR-9. An **Account Management Panel** is accessible via a settings/gear icon in the global Account Selector header component. It lists all accounts (active and inactive) showing `display_name`, `broker_type` badge, `is_paper` indicator, and `is_active` state.

FR-10. The panel includes an **Add Account** form with the following fields:
- `display_name` (text, required)
- `broker_type` (dropdown: Alpaca / IBKR, required)
- `is_paper` (toggle, default on)
- Dynamic credential fields based on `broker_type`:
  - Alpaca: `api_key`, `api_secret` — both rendered as `<input type="password">`
  - IBKR: `consumer_key`, `access_token`, `access_token_secret` — rendered as `<input type="password">`; `ibkr_account_id` — plain text
On submit, calls `TradingService.RegisterBrokerAccount`. On success the account list refreshes in context and the new account is auto-selected.

FR-11. Each active account row in the panel has a **Remove** button. Clicking it opens a confirmation modal ("Deregister `<display_name>`? In-flight orders will complete but no new orders can be placed."). On confirm, calls `TradingService.DeregisterBrokerAccount`. On success, if the removed account was the active selection, the selector falls back to the first remaining active account (or clears selection if none remain).

FR-12. Credential values are never held in browser state after a successful form submission — controlled inputs are cleared on success and on navigation away from the form. Credential fields use `<input type="password">` to suppress browser autofill display and prevent shoulder surfing.

## Out of Scope

- Server-side portfolio aggregation across accounts (client-side summation is acceptable).
- Per-account order history filtering beyond what `ListOrders` with `account_id` already supports.
- `xstockstrat-config-ui` — no account-related config changes.
- Mobile / responsive layout changes beyond what the existing design system provides.
- **SSL/TLS enforcement**: credential data travels over the existing HTTP transport. In production (DO App Platform), HTTPS is provided by the platform automatically. For self-hosted deployments, placing the app behind a TLS-terminating reverse proxy (nginx, Caddy) or attaching a certificate to the DO app spec are the intended hardening path — no application changes are required to support either. This is a deployment concern, not a code change.

## Acceptance Criteria

1. Account Selector in the global header renders all active accounts from `ListBrokerAccounts`; each shows `display_name`, a broker type badge (Alpaca / IBKR), and a paper/live indicator.
2. Selecting an account scopes the Place Order form, Portfolio panel, and Positions table to that `account_id`; all Connect-RPC calls include the correct `account_id`.
3. Placing an order without selecting an account is blocked in the UI with a clear error state.
4. The Portfolio panel shows one card/tab per account from `ListPortfolios`, each with that account's positions and P&L.
5. Switching accounts triggers a re-fetch of positions and portfolio data without a full page reload.
6. Inactive accounts are not selectable in the Account Selector.
7. `xstockstrat-insights` defaults to the first account on load; "All Accounts" option aggregates client-side; selection persists in URL state.
8. The Account Management Panel (gear icon) lists all accounts including inactive ones.
9. Submitting the Add Account form calls `RegisterBrokerAccount`; on success the new account appears in the selector and is auto-selected; credential fields are cleared.
10. Clicking Remove on an account shows a confirmation modal; on confirm calls `DeregisterBrokerAccount`; if removed account was active, selector falls back to the first remaining active account.
11. Credential inputs are rendered as `<input type="password">`; no credential values remain in React state after successful submission.
12. `pnpm run lint` and `pnpm run test:coverage` pass on both `xstockstrat-trader` and `xstockstrat-insights`.
13. No direct DB access from either frontend — all data fetched through Connect-RPC.
